# Workspaces

Per-project knowledge bases managed by the zeroone ecosystem.

## What is a Workspace?

A workspace is a directory under `workspaces/{project}/` that holds structured context for a specific project. It is the **static, file-based** layer of project knowledge — human-readable markdown that agents can read directly without querying Qdrant.

Workspaces complement Qdrant memory: files hold persistent, curated knowledge; Qdrant holds semantic search over conversation history and dynamic agent memories.

## Directory Convention

```
workspaces/
└── {project}/
    ├── context.md       # Project overview: stack, repo URL, conventions, team
    ├── decisions.md     # Architecture Decision Records (lightweight ADR log)
    └── runbook.md       # Operational runbook: how to start, stop, debug, deploy
```

Each project gets exactly one directory. The directory name matches the project identifier used in Mem0 scopes (`user_id="team:{project}"`).

## How Agents Discover Workspaces

Agents locate workspaces via the `ZEROONE_HOME` environment variable:

```bash
# In your shell profile
export ZEROONE_HOME="$HOME/path/to/zeroone"

# Agents read workspace files at:
$ZEROONE_HOME/workspaces/{project}/context.md
$ZEROONE_HOME/workspaces/{project}/decisions.md
$ZEROONE_HOME/workspaces/{project}/runbook.md
```

If `ZEROONE_HOME` is not set, agents fall back to discovering the zeroone repo:

```bash
find ~ -name "zeroone" -maxdepth 5 -type d 2>/dev/null | head -1
```

Set `ZEROONE_HOME` in your shell profile to make discovery deterministic and fast.

## What Goes in a Workspace vs Qdrant

| Content type | Location | Why |
|---|---|---|
| Project overview, stack, team | `context.md` | Stable, rarely changes, needs to be readable by humans |
| Architecture decisions | `decisions.md` | Curated log, benefits from git history |
| Operational runbook | `runbook.md` | Stable procedures, referenced during incidents |
| Agent conversation memories | Qdrant | Dynamic, semantic, high volume |
| Task outcomes and decisions | Qdrant (Mem0) | Per-agent or per-project scope, searchable |
| Temporary working context | Qdrant (Mem0) | Lives and dies with the task |

**Rule of thumb:** if you would commit it to git and want a human to read it, it belongs in the workspace. If it is generated by an agent during a session and searched semantically, it belongs in Qdrant.

## Creating a Workspace

Use the `zeroone` agent to create a workspace for a new project:

```
claude --agent zeroone
> setup my-project
```

This creates the workspace directory structure and provisions a Qdrant collection named `my-project` with the correct vector configuration for `nomic-embed-text` embeddings (768 dimensions, Cosine distance).

## Workspace Files Reference

### context.md

Document the project at a glance:

```markdown
# {Project Name}

## Overview
One-paragraph description of what this project does.

## Stack
- Language: Python 3.12
- Framework: FastAPI
- Database: PostgreSQL 16
- Repo: https://github.com/org/repo

## Conventions
- Branch naming: feat/, fix/, refactor/, chore/
- Commit style: conventional commits
- Tests: pytest, coverage >= 80%

## Team
- Owner: {name}
- Agents: neo (dev), trinity (finisher), oracle (orchestrator)
```

### decisions.md

Lightweight ADR log — one entry per decision:

```markdown
# Architecture Decisions

## 2026-04-10 — Use Qdrant for vector storage
**Status:** Accepted
**Context:** Need semantic search over agent memories.
**Decision:** Qdrant over Chroma — better performance at scale, Docker-native.
**Consequences:** Requires infra/docker-compose.yml to be running.
```

### runbook.md

Operational runbook for the project:

```markdown
# Runbook: {Project Name}

## Start local environment
```bash
cd infra && docker compose up -d
ollama serve  # if not already running
```

## Run tests
```bash
pytest tests/ -v
```

## Deploy
```bash
git push origin main  # triggers CI
```
```
