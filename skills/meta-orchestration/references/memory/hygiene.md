# Memory Hygiene

## Keep vs Prune Criteria

### Keep
- User preferences and feedback (unless contradicted by newer feedback)
- Active project context (deadlines, goals, constraints)
- External references that are still valid
- Decisions and their rationale

### Prune
- Completed tasks/initiatives (the code is the record)
- Stale project context (past deadlines, resolved incidents)
- Duplicates (merge into one canonical memory)
- Memories contradicted by newer information (update, don't keep both)

## Cleanup Protocol
1. Periodically review memories (monthly or on project milestones)
2. Verify references still exist/are valid
3. Merge duplicates
4. Archive completed project memories
5. Update stale information

## Anti-patterns
- Storing everything (noise overwhelms signal)
- Never pruning (stale memories cause wrong decisions)
- Storing code snippets (code changes, memory doesn't)
- Storing git history summaries (git log is authoritative)
